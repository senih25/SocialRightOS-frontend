# M2.2 Runtime Normalization Root Cause

## Olay

`P0-RUNTIME-001`, boş veya biçimsiz bir `POST /api/v1/eligibility-check` gövdesinin production runtime'da HTTP 500 üretmesidir. Başlangıç tabanı `74504420d5292c8e5e46811da64e13ba926eb3aa` üzerinde yerel yeniden üretim yapıldı.

## Kesin kök neden

```text
ROOT_CAUSE_FILE=src/lib/local-api-fallback.ts
ROOT_CAUSE_FUNCTION=buildLocalApiPayload -> buildGss -> decisionId
ROOT_CAUSE_LINE=30 (incident baseline)
THROWN_ERROR_TYPE=TypeError
UNSAFE_ASSUMPTION=Object olan her payload geçerli EligibilityCheckRequest ve eksik/unsupported benefit_code TR_GSS kabul edilebilir.
```

Baseline'da `typeof payload === "object" && payload !== null` kontrolü array'i de kabul ediyordu. `{}` ve `[]`, `switch (request.benefit_code)` içindeki `default` koluyla `buildGss` fonksiyonuna ulaşıyor; `decisionId(payload.benefit_code, ...)` daha sonra `undefined.toLowerCase()` çalıştırıyordu.

Minimal kanıt:

| Girdi | Baseline sonucu |
|---|---|
| `{}` | `TypeError: Cannot read properties of undefined (reading 'toLowerCase')` |
| `[]` | Aynı TypeError |
| `null` | Yanlış sınıflandırılmış 503 `unsupported_route` |
| JSON string | Yanlış sınıflandırılmış 503 `unsupported_route` |

## Neden daha önce yakalanmadı?

Mevcut local fallback testleri yalnız geçerli GSS, geçerli yaşlılık ve gelir testi happy-path'lerini kapsıyordu. Request parsing, array/root shape ve eksik zorunlu alanlar için HTTP düzeyi negatif matris yoktu.

Bu davranış production'a özgü değildi. Production denemesi, genel erişilebilir endpoint'e doğrudan boş JSON göndererek daha önce test edilmemiş runtime yolunu görünür kıldı. Derlenmiş yerel production sunucusu da düzeltme öncesi çağrı zincirinin aynısını kullanır.

## Kapsam sınırı

Endpoint, payload tipi, assessment sözleşmesi, CTA, analytics ve UI değiştirilmedi. Backend sözleşmesi yeniden tasarlanmadı.
