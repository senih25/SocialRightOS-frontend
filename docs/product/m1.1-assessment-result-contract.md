## Contract decisions

### Separate evidence origins

`AssessmentResult` has three independent collections:

- `reasons`: decision explanations from backend reasons or local legacy reasons,
- `ruleCriteria`: evaluated rule outputs,
- `missingInformation`: facts that were absent or unresolved.

Every item carries an explicit `origin` discriminator.

### Separate identifiers

The identity model distinguishes:

- `localEvaluationId`: generated by the frontend or local evaluator for one local assessment attempt,
- `backendDecisionId`: returned only by the eligibility backend,
- `requestId`: correlation identifier from the backend request/response flow.

A local legacy assessment must not invent a backend decision ID.

### No raw backend response in the persistent contract

The full `EligibilityCheckResponse` is not embedded in `AssessmentResult`.

Adapters return an `AssessmentAdapterValidationCarrier` containing:

- normalized `result`,
- temporary `sourcePayload` for parity tests and migration diagnostics.

The carrier is explicitly ephemeral. Persisted or UI-facing code should retain only `result`.

### Backward-compatible status model

M1.1 retains the current statuses without changing copy or behavior:

- `ELIGIBLE`
- `NOT_ELIGIBLE`
- `NEEDS_INFO`

### Benefit-specific extensions

Birth-grant payment details remain in the `TR_BIRTH_GRANT` extension. Legacy home-care income-limit context remains in the `TR_HOME_CARE_ALLOWANCE` extension. Other benefits currently use empty extensions.

## Passive adapters

- `adaptEligibilityResponse`
  - normalizes the existing backend response,
  - validates the backend benefit code,
  - keeps reason, rule, and missing-information origins separate,
  - returns the raw response only through the validation carrier.
- `adaptLegacyHomeCareResult`
  - preserves current status and reason strings,
  - sets `backendDecisionId` and `requestId` to `null`,
  - does not infer missing facts or rule results,
  - records the existing income-limit context without changing the threshold.

## Result-matrix test plan

The initial passive tests verify:

1. local and backend identifiers remain distinct;
2. reasons, rule criteria, and missing information remain separate;
3. record and array rule-result formats normalize consistently;
4. the raw backend response is absent from `AssessmentResult`;
5. birth-grant benefit details survive normalization;
6. unknown benefit codes fail closed;
7. local home-care results do not invent backend identifiers;
8. existing local home-care threshold context is preserved without changing behavior.

Future behavior-parity expansion should cover all three statuses for all four benefit codes and the complete existing home-care result matrix before any UI consumes this contract.

## Migration boundary

- no imports from `src/app/**`,
- no page consumes `AssessmentResult`,
- no existing response renderer changes,
- no route or anchor changes,
- no threshold or policy changes,
- no user-facing copy changes.
