# M1.2 TR_GSS Pilot Integration

## Scope

This pilot connects the existing GSS result flow to the passive assessment presentation contract.
It does not change eligibility decisions, routes, endpoints, payloads, analytics, thresholds, or
approved user-facing copy.

## Flow

`EligibilityCheckResponse` → `adaptEligibilityResponse` → `AssessmentResult<"TR_GSS">` →
`buildAssessmentPresentationModel` → GSS pilot adapter → existing inline result UI.

The raw response remains outside the presentation view-model. Presentation outcome is a UI
classification and never replaces the backend status.

## Preserved contracts

- Route: `/gss-gelir-testi`
- Endpoint: `/api/v1/eligibility-check`
- Request payload: `buildGssPayload`
- Analytics event names and payload shape
- Loading and API error paths
- Existing title, summary, disclaimer, badge/status labels, CTA labels and destinations

## Accessibility acceptance

- Result updates use a polite live region.
- Status remains available as text and is not conveyed by color alone.
- Reasons, rule criteria, and missing information remain separate semantic collections.
- Empty collections are omitted.
- Existing heading hierarchy, contextual links, keyboard behavior, and mobile layout remain intact.
- Technical identifiers and mapper failure details are not rendered.

## Regression gates

Targeted GSS tests run first, followed by lint, the complete test suite, production build with
TypeScript, `git diff --check`, allowlist verification, and a clean-worktree check after commit.

## Out of scope

Other benefit integrations, shared UI rewrites, new test tooling, observability changes, nested
birth-grant extension hardening, deployment, Cloud Console, Cloud Shell, domain, Firecrawl, and T03.
